# TON Wallet Tracker

Telegram-бот, который отслеживает TON-кошельки и присылает уведомления о транзакциях
(покупки, продажи, свопы, переводы, mint/burn, стейкинг, NFT) с минимальной задержкой.

> **Это намеренно компактный MVP**, а не микросервисная платформа на 100k пользователей.
> Архитектура рассчитана на десятки-сотни пользователей и тысячи кошельков на одном
> VPS (2 ядра / 4 ГБ). Раздел [«Масштабирование»](#масштабирование) объясняет, что
> менять при росте — но без преждевременного усложнения.

---

## Почему такой подход к детекту (выбор провайдера)

Главное требование — **самая низкая задержка уведомления**. Сравнение вариантов:

| Подход | Задержка | Сложность | Вывод |
|---|---|---|---|
| **TonAPI SSE/WebSocket** | ~доли секунды (push) | низкая | ✅ **выбрано** |
| TonCenter polling | секунды (опрос) | средняя | медленнее, больше запросов |
| LiteServer / ADNL (tonlib) | низкая | высокая | свой парсинг бочей, дорого по времени |

**Решение:** TonAPI **Streaming SSE** как мгновенный триггер + REST `/v2/accounts/{addr}/events`
как источник правды с уже **разобранными действиями** (`JettonSwap`, `JettonTransfer`,
`TonTransfer`, `NftItemTransfer`, …). Это избавляет от написания собственных DEX-декодеров
и даёт push-задержку. Дедупликация — через курсор `last_lt` в Postgres.

Документация: <https://docs.tonconsole.com/tonapi/streaming-api>

## Архитектура

```
                ┌─────────────────────────── один Node-процесс ───────────────────────────┐
 TonAPI SSE ───►│  TonStream ──► Watcher ──► classify ──► fan-out ──► Notifier ──► grammY  │──► Telegram
 (push)         │   (триггер)    │  cursor    (buy/sell)   подписчики   throttler+retry     │
                │                ▼                                                          │
                │           PostgreSQL (users, wallets, subscriptions, groups, stats)       │
                └──────────────────────────────────────────────────────────────────────────┘
                        ▲ sweep раз в N сек — страховка от пропусков SSE
```

**Принципы:**
- **Один кошелёк — один вотчер глобально.** Уникальный адрес = одна строка в `wallets`,
  SSE подписан на множество уникальных адресов, событие веером рассылается всем подписчикам.
- **Дедупликация и защита от повторов на рестарте:** курсор `last_lt` в `wallets`.
  Уведомляем только события с `lt > last_lt`.
- **Без блокировки event loop:** всё async, обработка одного кошелька защищена от гонок.
- **Надёжность:** ретраи REST с backoff, авто-reconnect SSE, троттлинг и авто-ретрай Telegram
  (FloodWait), graceful shutdown.

## Возможности

**Пользователь:** список кошельков, добавление (одиночно/списком), удаление, метки,
группы, фильтры по типам событий, импорт/экспорт (JSON/CSV/TXT), статистика, помощь.
Всё на inline-клавиатурах. Дубликаты адресов запрещены.

**Уведомление содержит:** тип события, токен/символ/контракт, сумму, объём в TON,
хэш, ссылку на эксплорер, задержку детекта. Кнопки: **Dtrade**, **Explorer**, **Копировать контракт**.

Формат Dtrade-ссылки: `https://t.me/dtrade?start=rifle_<КОНТРАКТ>`

**Админ** (`ADMIN_ID` из `.env`): дашборд, пользователи, блокировка, статистика,
здоровье системы (SSE, задержка, события/мин, RAM, аптайм), рассылка. Команда `/admin`.

## Установка и запуск

### Вариант A — Docker (рекомендуется для VPS)

```bash
git clone <repo> && cd tracker-bot
cp .env.example .env
nano .env            # впишите BOT_TOKEN и TONAPI_KEY
docker compose up -d --build
docker compose logs -f app
```

Контейнер `app` сам прогоняет миграции перед стартом. Postgres хранит данные в volume `pgdata`.

### Вариант B — локально (разработка)

Нужен запущенный PostgreSQL.

```bash
npm install
cp .env.example .env        # DATABASE_URL -> localhost
npm run db:migrate          # применить схему
npm run dev                 # запуск с авто-перезагрузкой
```

### Команды

```bash
npm run dev         # разработка (tsx watch)
npm run build       # сборка в dist/
npm start           # прод-запуск
npm run typecheck   # tsc --noEmit
npm test            # vitest
npm run db:generate # сгенерировать SQL-миграцию из схемы
npm run db:migrate  # применить миграции
```

## Конфигурация (`.env`)

| Переменная | Описание |
|---|---|
| `BOT_TOKEN` | токен бота от @BotFather |
| `ADMIN_ID` | Telegram ID администратора (по умолчанию `5453071067`) |
| `TONAPI_KEY` | ключ TonAPI с <https://tonconsole.com> (обязателен для прода) |
| `TONAPI_BASE_URL` | базовый URL TonAPI (по умолчанию `https://tonapi.io`) |
| `DATABASE_URL` | строка подключения PostgreSQL |
| `LOG_LEVEL` | уровень логов pino (`info` по умолчанию) |
| `SWEEP_INTERVAL_SEC` | период страхующего опроса кошельков (сек, по умолчанию 120) |
| `WATCHER_CONCURRENCY` | лимит параллельных проверок кошельков/TonAPI (по умолчанию 8) |
| `TONAPI_TIMEOUT_MS` | таймаут REST-запросов к TonAPI (по умолчанию 10000) |
| `TONAPI_RETRIES` | число повторов TonAPI при 429/5xx/сетевых ошибках (по умолчанию 3) |

Секреты только в `.env`, в коде их нет.

## Масштабирование

Текущий MVP — один процесс + Postgres. Что менять **по мере роста** (и не раньше):

1. **Много кошельков (> ~неск. тысяч на коннект).** TonAPI SSE — один URL со списком
   адресов; при большом числе перейти на **WebSocket** (`subscribe_account`, до 1000
   аккаунтов на соединение) и шардировать адреса по нескольким коннектам.
2. **Пиковая нагрузка на отправку.** Вынести рассылку в очередь (BullMQ + Redis) и
   несколько воркеров; сейчас хватает встроенного троттлера grammY.
3. **Несколько инстансов.** Сделать вотчер отдельным процессом от бота; шардировать
   кошельки по `wallet_id % N`. Бот тогда только UI, вотчер(ы) — детект.
4. **БД.** Индексы уже расставлены (`subscriptions.wallet_id`, уникальные ключи).
   При росте — реплики на чтение, партиционирование лога событий (если появится).

Это сознательно отложено: добавлять раньше реальной потребности — переусложнение.

## Тестирование

`npm test` покрывает чистую бизнес-логику: валидацию адресов, классификацию
buy/sell/swap/переводов, форматирование сумм, генерацию Dtrade-ссылки и рендер уведомления.
Хендлеры бота и сетевые вызовы намеренно не мокаются — они тонкие, основная логика в чистых функциях.

## Известные упрощения (осознанные)

- **Long polling** вместо вебхука — для одного инстанса проще и без публичного HTTPS.
- **In-memory сессии** — диалоговое состояние теряется при рестарте (некритично).
- **Redis не используется** — дедуп и кэш решены через Postgres-курсор и память процесса.
- **Ликвидность** детектируется по эвристике из `simple_preview` (TonAPI не всегда даёт
  отдельный тип) — best-effort.
